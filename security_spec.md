# Спецификация безопасности Firestore (Security Rules Spec)

Данный документ описывает инварианты доступа, сценарии возможных верификационных угроз и оборонительные правила для Fitness Verification PWA.

## 1. Инварианты данных
- **Профиль пользователя (`/users/{userId}`)**: Пользователь может читать любой профиль, но изменять свой профиль может только сам владелец. Поле `uid` должно строго соответствовать `request.auth.uid`. Пользователи не могут самостоятельно изменять свой статус подписки `subscriptionStatus` и `points` без прохождения согласования (подписка меняется только сервером/эмуляцией транзакций).
- **Тренировки/Подтверждения (`/submissions/{submissionId}`)**: Любой вошедший пользователь может просматривать список тренировок. Обычный пользователь может создавать запись тренировки только в статусе `pending` и только от своего лица (`userId == request.auth.uid`). Модификация полей `status` (pending -> approved/rejected) разрешена исключительно администраторам верстки (в эмуляции) или служебными триггерами.
- **Особые системные разделы**: `ranks` и `subscriptions` защищены от любых клиентских записей.

## 2. Сценарии атак (The Dirty Dozen Payloads)

| Сценарий атаки | Коллекция | Ожидаемый результат | Описание эксплоита |
|---|---|---|---|
| ID Poisoning | `/users` | `PERMISSION_DENIED` | Попытка внедрить ID с недопустимыми символами |
| Привилегии подписки | `/users/user123` | `PERMISSION_DENIED` | Анонимный апгрейд до Elite тарифа |
| Накрутка очков | `/users/user123` | `PERMISSION_DENIED` | Изменение points напрямую клиентом |
| Чужой профиль | `/users/user_other` | `PERMISSION_DENIED` | Перезапись чужого профиля другим UID |
| Фейк-авторство тренировки | `/submissions/sub_99` | `PERMISSION_DENIED` | Создание записи с `userId` администратора |
| Само-верификация | `/submissions/sub_99` | `PERMISSION_DENIED` | Попытка обновить статус с `pending` на `approved` |
| Повреждение метаданных рангов| `/ranks/rank_elite` | `PERMISSION_DENIED` | Попытка удалить или изменить шкалу достижений |
| Сброс логов верификации | `/verification_jobs/job1`| `PERMISSION_DENIED` | Очистка истории ML-анализа видео |

## 3. Модель проверок (Security Rules Test Matrix)
- Все тесты гарантируют возврат `PERMISSION_DENIED` при несовпадении `request.auth.uid` с ключевыми полями документов или при попытке обойти валидаторы.
